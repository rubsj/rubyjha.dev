## The Problem

Pre-trained embedding models like all-MiniLM-L6-v2 produce general-purpose sentence embeddings, but they don't understand domain-specific similarity. Two sentences that are semantically related in your domain might land far apart in embedding space, while unrelated sentences cluster together. Fine-tuning fixes this — but full fine-tuning updates all 22.7 million parameters, producing an 86 MB model that's expensive to train and deploy.

I wanted to answer a practical question: can LoRA (Low-Rank Adaptation) deliver comparable fine-tuning quality while training only a fraction of the parameters? If so, by how much does it reduce model size and training time? I built a head-to-head comparison using contrastive learning on sentence pairs, with a comprehensive evaluation suite to measure the gap precisely.

Both fine-tuning approaches start from the same base model (all-MiniLM-L6-v2) and train on identical sentence pairs using CosineSimilarityLoss. Standard fine-tuning updates all 22.7M parameters. LoRA injects low-rank adapter matrices into the attention layers, training only 73K parameters (0.32% of the model) while freezing everything else.

The evaluation suite measures three complementary metrics: Spearman correlation (ranking quality), AUC-ROC (classification quality), and Cohen's d (effect size between similar and dissimilar pair distributions). UMAP visualizations show the embedding space structure before and after fine-tuning, and everything renders into an interactive HTML report.

The baseline model was actively wrong — Spearman correlation of -0.219, AUC-ROC of 0.373, and a negative Cohen's d of -0.419. The similarity margin between positive and negative pairs was -0.083, meaning the model rated dissimilar sentences as _more_ similar than similar ones. It wasn't just bad; it was inverted.

Standard fine-tuning fixed everything: Spearman jumped to 0.853, AUC-ROC to 0.994, and Cohen's d to 7.727. The similarity margin flipped from -0.083 to +0.940. False positives dropped from 137 to 3 — a 97.8% reduction. These are production-grade numbers.

LoRA achieved 96.9% of standard's performance across all metrics — with only 0.32% trainable parameters (73K vs 22.7M). The LoRA adapter is 0.28 MB compared to the full model's 86.65 MB — a 309x reduction in model size. Training completed in 35 seconds on an M2 MacBook Pro versus 61 seconds for standard (43% faster). The project includes 112 tests at 100% pass rate, 8 comparison charts, and an interactive HTML report.

## Key Decisions

<Callout type="tip">
  **LoRA is production-viable for resource-constrained deployment.** At 96.9% of standard performance with a 309x
  smaller adapter, LoRA is the clear choice when you need to fine-tune for multiple domains or deploy on
  resource-limited infrastructure. The 3.1% quality gap is measurable but acceptable for most applications.
</Callout>

<Callout type="info">
  **QLoRA skipped — quantization unnecessary at this scale.** QLoRA combines LoRA with 4-bit quantization for models
  that don't fit in memory. all-MiniLM-L6-v2 is only 86 MB — it fits comfortably on any modern machine. Adding
  quantization would introduce precision loss for zero practical benefit.
</Callout>

<Callout type="info">
  **CosineSimilarityLoss for contrastive learning.** For binary sentence pair data (similar/dissimilar),
  CosineSimilarityLoss directly optimizes the cosine similarity between embeddings. It's simpler than triplet loss (no
  need for negative mining), converges reliably, and maps directly to the similarity metric used at inference time.
</Callout>

## The LoRA Learning Rate Discovery

<Callout type="warning">
  **LoRA needs 10x higher learning rate — and nobody tells you this.** My first LoRA attempt used the same learning rate
  as standard fine-tuning (2e-5) and produced garbage results — metrics barely moved from baseline. The fix was bumping
  the learning rate to 2e-4. LoRA's low-rank adapter matrices have far fewer parameters absorbing the gradient signal,
  so they need a more aggressive learning rate to make meaningful updates. This 10x multiplier isn't documented anywhere
  obvious in the PEFT or sentence-transformers docs. I only found it through systematic hyperparameter search after the
  initial failure.
</Callout>

This was the most valuable lesson from the project. If I'd only run LoRA with the default learning rate, I would have concluded that LoRA doesn't work for sentence transformer fine-tuning — a completely wrong conclusion caused by a single hyperparameter.

<Callout type="warning">
  **Adapter merge is a silent failure mode.** LoRA achieved Spearman 0.827 during training, but post-training evaluation
  produced baseline-identical metrics due to an adapter merge issue during inference. The `PeftModel.from_pretrained` +
  `merge_and_unload` path didn't correctly merge adapter weights before encoding. The model runs without errors but
  produces unmodified embeddings — a real production risk with LoRA deployments that's invisible without comprehensive
  evaluation metrics.
</Callout>
